## Augment `review_roadmap` / `ticket-writer`

### [ ] Label state machine for GitHub issues (from `github-triage`)

- Add four state labels: `needs-triage`, `ready-for-agent`, `ready-for-human`, `wontfix`
- Enforce: every issue has exactly one state label + one category label (`bug` / `enhancement`)
- `ready-for-agent` = AFK counterpart. Bridges the HITL/AFK body tag into the issue list view so you can filter on it.
- **Where:** `ticket-writer` SKILL.md (document the convention) + `review_roadmap` command (enforce during roadmap reviews)
- **Effort:** ~30 min (labels + skill update + one-time backfill)

### [ ] Mandatory AI-disclaimer on auto-generated issue comments (from `github-triage`)

- Pattern: every AI-generated comment prefixed with `> *This was generated by AI during triage.*`
- Applies when Claude comments on issues via `gh issue comment`
- **Where:** `ticket-writer` SKILL.md rules section
- **Effort:** ~5 min

### [ ] "Durability rule" for ticket bodies (from `qa`)

- Rule: issue bodies must remain comprehensible after major refactors. No file paths, no line numbers, no code locations.
- Tickets describe user-facing behavior and domain concepts, not implementation.
- **Where:** `ticket-writer` SKILL.md — add to "Anti-Patterns" section
- **Effort:** ~10 min

## Augment `create_plan`

### [ ] "Always-shippable commits" rule for refactor plans (from `request-refactor-plan`)

- For refactor-category plans specifically: break into minimal commits where *the program always works between commits*.
- Each commit is independently verifiable (`make check` passes, tests pass).
- Stronger invariant than phases: you can stop/resume at any commit boundary.
- **Where:** `create_plan.md` — add a conditional rule: "For refactors, define commit boundaries inside each phase such that every commit leaves the system in a working state."
- **Effort:** ~10 min

## Augment `brainstorm`

### [ ] Require recommended answer for every question (from `grill-me`)

- Rule: every question Claude asks the user during brainstorming comes with a proposed answer.
- Current `brainstorm` asks one-at-a-time but doesn't require proposing an answer.
- **Why:** Users answer faster when given a starting point. Reveals Claude's current thinking for calibration.
- **Where:** `brainstorm.md` — tighten the "One question at a time" rule.
- **Effort:** ~5 min

### [ ] Parallel sub-agent interface generation for greenfield designs (from `design-an-interface`)

- Pattern: when designing a new module/component from scratch, spawn 3+ sub-agents with *radically different* design constraints (minimize methods / maximize flexibility / optimize common case).
- Already captured in `improve-architecture` for existing code; this would be the greenfield analogue in `brainstorm`.
- **Where:** `brainstorm.md` — add a branch: "If designing a new component or API, consider /improve_architecture-style parallel design exploration."
- **Effort:** ~15 min
