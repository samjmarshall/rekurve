# Technical Task Template

> Body principles (durability, behaviour-not-files, AC, out-of-scope) are in **[agent-brief.md](agent-brief.md)** — every ticket body must satisfy them. This template gives the type-specific structure on top.

Use this template for infrastructure work, refactoring, technical debt, non-user-facing improvements, and engineering improvements.

## Template Structure

```markdown
Title: [Action verb] [technical component] [purpose]
Example: Add database index on orders.created_at for reporting performance

## Purpose
[Why this technical work is needed]

## Context
**Current situation**: [What's the problem?]
**Proposed solution**: [What are we doing?]
**Why now**: [Why is this a priority?]
**Impact**: [Performance, reliability, scalability benefit]

## Technical Approach
[High-level plan — enough guidance without over-prescribing]

## Acceptance Criteria
See `agent-brief.md` § Acceptance criteria for the full rules.

- [ ] Composite index on `orders(created_at DESC, customer_id)` exists in production.
- [ ] Reporting query latency drops from ~45s to <5s on the staging dataset.
- [ ] Existing queries continue to return identical result sets.
- [ ] `make check` passes.

## Schema changes (if applicable)
Describe the change behaviourally — new columns, new constraints, index target, expected query plan improvement. Do not embed hand-crafted SQL. Migrations are generated by `make db_generate` during implementation; the ticket body should not pre-empt that.

## Testing Plan (optional — risky changes only)
For migrations, performance work, or anything that could regress unrelated areas, describe the verification steps beyond `make check` — e.g. baseline + target benchmark, regression suite to run, staging soak time. Skip this section entirely for low-risk tasks.

## Dependencies
- [ ] Staging environment available for testing.
- [ ] Deployment window scheduled.

## Resources
- Query performance baseline: [dashboard link]
- Related performance work: [ticket references]
```

## Guidelines

**Title verbs (this type):** Add, Refactor, Migrate, Upgrade, Optimize.
Example: "Add database index on orders.created_at for reporting performance".

**Project-specific notes:**
- Schema changes: describe behaviourally; SQL is generated via `make db_generate` during implementation, never hand-crafted in the body.
- For risky changes, always include a Testing Plan with baseline + target.
