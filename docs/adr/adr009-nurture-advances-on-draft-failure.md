# Nurture scheduler advances `nextStepAt` even when `draftMessage` throws

**Status:** superseded by [adr011](adr011-followup-drafts-retry-then-pause.md) (proposed 2026-05-04)

`runSchedulerTick` walks every active `nurture_sequences` row whose `nextStepAt` has passed and, for each one, attempts a `draftMessage` call. On success, the draft is inserted into `message_queue` with `status='pending'` and `drafted++`. On failure (Anthropic 5xx, timeout, schema-validation reject, `INSERT INTO message_queue` rejection, any throw inside the per-row `try` block), the error is logged with `[nurture-scheduler] draftMessage failed for sequence <id>:` and `failed++` — and `nextStepAt` is *still* advanced by the cadence (`scheduler.ts:125-132`). One Claude failure costs one missed touch on that lead's cadence boundary, never more. The next tick (24h later) sees the row with a future `nextStepAt` and skips it; the lead picks up its sequence again on the *next* cadence boundary (`+CADENCE_DAYS[sequenceType]` from the failed tick). The rule prioritises bounded forward progress over guaranteed delivery: a poison row that consistently throws cannot block every later row in the same tick, cannot accumulate into a backlog, and does not need a retry surface to recover.

## Considered options

- **Keep `nextStepAt` unchanged on failure; let the next tick re-attempt the same row.** Rejected for two compounding reasons. First, poison rows block the tick: a row whose lead has a malformed field that consistently throws inside `draftMessage` (e.g. a Zod validation reject on a missing required input) would be re-selected every tick forever, sitting at the front of the due-rows list (oldest `nextStepAt` first) and burning the prompt-cache prefix on a row that will fail. Second, backlogs grow: a multi-hour Anthropic outage spanning the tick window would leave every due row unadvanced; the next day's tick now has yesterday's backlog plus today's due rows, and a second outage compounds. The system's failure mode shifts from "miss one touch" to "unbounded queue depth," which is a much worse posture for a daily-cron architecture. The recovery contract is also worse: with "advance regardless," `failed` from the cron response equals `missed touches` exactly; with "leave unchanged," the operator has to query the DB to find rows whose `nextStepAt` is in the past after a tick ran.
- **Add a `failed_count` column and a retry-with-backoff sweep.** Rejected for pilot. At pilot scale (one consultant, single-digit due rows per tick), Claude-outage-on-this-row is a once-a-quarter event. Building backoff + dead-letter + replay tooling + a "reset failed_count" UI for the operator is multi-day work whose value is bounded by an event we have no evidence happens. It also introduces two cadence systems — the retry cadence (`+1h`, `+4h`, `+24h`, …) and the sequence cadence (`+3d`/`+7d`/`+14d`) — and reasoning about "when will this lead next be touched?" requires understanding both, with edge cases (a retry that succeeds *after* the next sequence cadence boundary would have fired) needing an explicit precedence rule. The current rule has one cadence, period. If pilot evidence shows real touch-loss, the durable-retry investment goes into the Vercel Workflows spike (#153), not a hand-rolled column.
- **Wrap the per-row `draftMessage` + `INSERT message_queue` + `UPDATE nextStepAt` in a single transaction; on draft failure, leave the row's cursor unchanged.** Rejected on three grounds. It collapses to option 1 — a draft failure rolls back the `nextStepAt` advance, so the row is re-selected next tick, with the same poison-row and backlog problems. There is no external transaction boundary to gain from: `draftMessage` is an Anthropic HTTP call and cannot be inside a Postgres transaction; the "transaction" would only wrap the two DB writes, which already commit sequentially per `await`. And it costs concurrency — holding a row-level lock on `nurture_sequences` across an Anthropic call (~1–3 seconds) makes a future "parallelise the tick" change strictly harder. We are sequential by design today (prompt-cache reasons, see feature-doc trade-off), but locking the assumption into the schema is bad value.

## Consequences

- **The cost of any single Claude failure is bounded and known.** One missed touch on the failed lead's cadence boundary (`+3` / `+14` / `+7` days, depending on `sequenceType`). Never more than one — the next tick advances past the failed row, so a second tick on the same row would only fire after another full cadence period. The operator can read `{ drafted, failed }` from the cron response in Vercel logs and reason about touch-loss exactly: `failed * 1 missed touch each`. This bounded-cost property is the core trade and must not be weakened by future changes (e.g. adding a "retry within tick" loop) without revisiting this ADR.
- **A poison row is invisible until its lead's queue stays quiet across multiple cadences.** A row whose lead consistently throws inside `draftMessage` (malformed lead data, prompt that triggers a schema-validation reject, etc.) will fail-and-advance on every tick that catches it due — `failed` increments in the cron response, but no per-row state records the streak. The lead silently never gets a draft. The signal is the consultant noticing this lead's action queue stays empty. Mitigation lives outside this ADR (a future "leads with N consecutive failures" report would close the gap); the rule itself accepts the trade.
- **Anthropic outages spanning a full tick window cost one touch per active sequence, not a backlog.** If Claude is down for the entire tick, every due row's `nextStepAt` advances by its cadence and the next day's tick is back to normal load. There is no compounding effect — this is the central operational benefit over option 1. The cost is concentrated on the day of the outage and recovers automatically.
- **The `INSERT INTO message_queue` failure path inherits the same swallow-and-advance rule, but the diagnostic signal is different.** A queue-insert throw indicates schema/migration drift in `message_queue`, not a Claude outage. Both surface as `[nurture-scheduler] draftMessage failed for sequence <id>:` (the log message is shared because both live inside the same `try`), so a future operator chasing a `failed` count cannot tell the two apart from logs alone — they have to inspect the inner error. If Claude-outage and DB-write failures need to be distinguished operationally, the log line splits into two before this ADR is revisited; the rule itself does not change.
- **The decision composes with adr007's "trust the 202" posture.** Both ADRs accept a known silent-failure surface in exchange for not building a retry/reconciliation pipeline pre-PMF. ADR-007 swallows SMTP bounces (`#154`); this ADR swallows Claude failures. The combined posture is: at pilot scale, the cost of building durable retry surfaces exceeds the cost of the touches/sends those retries would save, and the work is deferred until pilot evidence justifies it. A future ADR that introduces durable retry (Vercel Workflows spike #153 is the leading candidate) likely supersedes both this ADR and adr007's bounce gap simultaneously — they share a successor, not a constraint.
