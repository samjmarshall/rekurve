# Outlook send with HubSpot BCC reconciliation

**Status:** accepted

Outbound email approved on `/dashboard` is sent via Microsoft Graph `/me/sendMail` on the consultant's own M365 mailbox, with `env.HUBSPOT_BCC_ADDRESS` (the portal-specific `bcc-NNNNN@bcc.hubspot.com`) BCC'd on every send. HubSpot ingests the BCC, creates the timeline engagement asynchronously, and fires `object.creation` for `objectTypeId = "0-49"`; `handleEmailCreation` matches the engagement back to the local `conversations` row by lead + delivery method + Â±5min `createdAt` window + matching subject and stamps `hubspotActivityId`. The lead sees `firstname.lastname@creationhomes.com.au` as the sender â€” Rekurve stays invisible infrastructure â€” and the consultant gets a real Outlook thread on their own machine that future replies will land on.

## Considered options

- **HubSpot Single-Send Transactional API** â€” what issue [#130](https://github.com/samjmarshall/www/issues/130) originally specified. Rejected because it requires Marketing Hub Enterprise + the Transactional Email add-on (~$400/mo) â€” cost-gated out of the pre-PMF pilot.
- **HubSpot Engagements API direct POST after Graph send** â€” call `POST /crm/v3/objects/emails` from the approve mutation once Graph returns 202, stamp `hubspotActivityId` synchronously, skip the BCC plumbing and the reconciliation window entirely. Rejected for two reasons. First, HubSpot's native BCC ingestion is the highest-fidelity way to populate the timeline â€” it produces an engagement with the right MIME, headers, threading, and direction flags because HubSpot built it for exactly this; a hand-rolled Engagements POST has to recreate that surface and will drift from "looks like a normal email in the timeline" in ways the consultant or sales manager will eventually notice. Second, the BCC mechanism is forward-compatible with inbound replies: when the lead replies and the consultant hits Reply in Outlook, the original BCC stays on the thread, so HubSpot picks the reply up the same way it picks up the outbound. Engagements API outbound + manual inbound handling is a much bigger lift later.
- **Resend from a shared `noreply@rekurve` domain** â€” already installed for OTP, simpler to wire. Rejected because the lead would see a Rekurve-branded sender, which contradicts the product's posture: the consultant is the relationship and Rekurve is invisible infrastructure. Also no Outlook thread, no Sent Items entry on the consultant's machine, no reply visibility in their normal inbox.

## Consequences

- **Consistent with ADR-003.** HubSpot owns the canonical timeline engagement; the local `conversations` row mirrors it via `hubspotActivityId` once HubSpot tells us what the engagement ID is. The app does not synthesise the timeline shape â€” same posture as contact data.
- **`conversations.hubspotActivityId` is nullable for ~15â€“90s after every send, and indefinitely if the webhook is dropped.** This is the operational shape of "HubSpot creates the engagement asynchronously." Most consumers (lead profile, queue UI, conversation log) tolerate the null state and should continue to. The approve mutation does not block on reconciliation.
- **Future features that require a guaranteed `hubspotActivityId` cannot be built on `conversations` without first solving missed-webhook recovery.** A nightly reconciler is the obvious shape (find rows older than ~15min with null `hubspotActivityId`, query HubSpot for matching engagements, stamp them) but is deliberately deferred until pilot evidence shows the webhook actually drops events. Any feature that *cannot* tolerate indefinite null on this column owns the reconciler as a blocker.
- **Single mailbox per consultant is locked in by schema.** `ms_graph_tokens.userId` is the primary key â€” one Microsoft account per `user.id`, no shared mailbox, no send-as, no delegated send. Multi-mailbox onboarding is a schema change and an OAuth-flow change, not a config tweak.
- **Subject-based reconciliation is fuzzy at the Â±5-minute boundary.** Two outbound emails to the same lead with the same subject within 5 minutes will race; the closest-by-`createdAt` candidate wins, the other stays null. Acceptable at pilot volume because AI drafts vary subjects; if collision rates rise, the matching window narrows or the match key extends (e.g. body hash) before the window widens.
- **Silent SMTP-bounce gap.** Graph returns 202 (queued) before SMTP delivery. Microsoft can later bounce with `550 5.7.501 Spam abuse detected from IP range` and the dashboard will already have shown "Sent via email". This is a known limitation of "trust the 202", tracked as [#154](https://github.com/samjmarshall/www/issues/154); the in-flight successor design [`thoughts/designs/2026-04-27-email-compose-providers.md`](../../thoughts/designs/2026-04-27-email-compose-providers.md) addresses it via webhook-driven send detection, and may supersede this ADR once it ships.
- **MSAL `acquireTokenByCode` workaround.** The SDK's typed result drops `refresh_token`, so the OAuth callback POSTs `/oauth2/v2.0/token` directly to capture it. If MSAL changes its surface, the hand-rolled fetch needs review â€” this is fragile against SDK upgrades and worth flagging in code review when `@azure/msal-node` is bumped.

This ADR records the shipped state. A successor design (`thoughts/designs/2026-04-27-email-compose-providers.md`) is under active consideration in response to [#154](https://github.com/samjmarshall/www/issues/154) (silent SMTP failure) and [#156](https://github.com/samjmarshall/www/issues/156) (MIME-content sendMail); if it ships, it will supersede this ADR.
