# Inngest Outbox Foundation Implementation Plan

## Context

Foundational infrastructure for [ADR-014](../../docs/adr/adr014-outbox-pattern-for-inngest-delivery.md) — a transactional outbox plus cron sweep that gives at-least-once delivery to Inngest. No call-site cuts over in this ticket; downstream tickets (#258–#261) migrate write paths once the surface exists. This is the gate every other ticket in milestone "Outbox: DB-First Lead Writes" is blocked on.

## Current State

- `drizzle-orm/neon-http` driver in `src/server/db/index.ts:8-10` — HTTP transactions are non-interactive (queries batched into one request, so `.returning()` mid-callback does not surface mid-tx).
- Existing cron uses Vercel cron + Bearer auth (`src/app/api/cron/nurture-scheduler/route.ts:11-14`); test pattern at `src/app/api/cron/nurture-scheduler/__tests__/route.test.ts:7-22` mocks `~/env`, `~/server/db`, and the worker module via `rs.doMock`.
- Logging convention is `console.error("[scope] ...")` (`src/server/api/routers/leads.ts:57`, `src/server/dispatch/__tests__/email-dispatch.test.ts:54-72`).
- Schema barrel re-exports per file (`src/server/db/schema/index.ts:1-9`); seeder TRUNCATE list lives in `scripts/seed-dev.ts:45-47`.
- No Inngest dependency, client, route handler, or function registry exists yet.
- `vercel.json` holds the nurture cron only.

## Desired End State

- `outbox` table exists in Postgres with the ADR-014 schema and partial index on unprocessed rows.
- `outbox.publish(tx, eventName, payload)` exists, returns the row id, and refuses to compile when the caller passes the bare `db` (no transaction).
- An Inngest app is wired (client + serve handler at `/api/inngest`), with two scheduled functions registered: a `* * * * *` sweep and a daily prune.
- Re-running the sweep is a no-op for already-delivered rows because `processed_at IS NULL` filters them out and Inngest treats the row id as an idempotency key.
- `src/server/outbox/README.md` documents the post-commit publisher pattern so call-site PRs do not have to reverse-engineer it.
- Existing flows (lead create/update, nurture scheduler) are unchanged — no consumer of `outbox.publish` ships in this PR.

## Out of Scope

- Cutting over `leads.create` / `leads.update` / nurture / Outlook / Twilio call sites (tickets #258–#261).
- BetterStack alert rules for stuck rows / `attempts > 10` (per ticket "How" section).
- Operator UI for inspecting stuck outbox rows (Inngest dashboard covers pre-PMF observability).
- Migration off `neon-http` to `neon-serverless` (call-site PRs will decide when transactional `.returning()` becomes blocking).
- Wiring Inngest cloud (event key / signing key are added as optional env vars; production secrets land later).

## Approach

Five sequential phases, each independently verifiable. Phase 1 lands the storage. Phase 2 stands up the Inngest app with an empty function set so the route handler compiles. Phase 3 adds the typed publish helper (the only thing call-site PRs touch). Phase 4 lands the sweep + prune Inngest functions and wires them into the serve handler. Phase 5 documents the publisher pattern and the make/dev workflow. Tests ship in the phase that introduces the behaviour they cover, per the project's TDD vertical-slice convention.

## Phase 3: `outbox.publish` helper + tests

### Changes

- `src/server/outbox/index.ts` — new module exporting `publish(tx, eventName, payload)`.
  - First arg is typed as `PgTransaction<NeonHttpQueryResultHKT, typeof schema, ExtractTablesWithRelations<typeof schema>>` so callers cannot pass the bare `db`.
  - Generates the row id via `crypto.randomUUID()` before the insert (the neon-http transaction is non-interactive, so a `.returning({ id })` would not surface the value mid-callback — generating client-side keeps the helper usable in transactional call sites).
  - `await tx.insert(outbox).values({ id, eventName, payload })` then `return id`.
  - `eventName` typed as `string` for now; a typed event-name union is a follow-up when the first call site lands.
- `src/server/outbox/__tests__/publish.test.ts` — rstest unit test, mirroring `src/server/dispatch/__tests__/email-dispatch.test.ts:32-72`-style chainable mock for `tx.insert(...).values(...)`.
  - Compile-time check: a `// @ts-expect-error` block calls `publish(db, ...)` proving `make check` rejects the bare `db`.
  - Runtime check: passes a mock tx, asserts `tx.insert` was called with `outbox`, that values include the generated `id`, and that the returned id matches.

## Phase 4: sweep + prune Inngest functions

### Changes

- `src/server/outbox/sweep.ts` — exports `outboxSweep`, an Inngest scheduled function created with `inngest.createFunction({ id: "outbox-sweep", triggers: { cron: "* * * * *" } }, async ({ step }) => { ... })`.
  - One `step.run("select-unprocessed", ...)` selecting up to 100 rows with `processed_at IS NULL AND created_at < now() - interval '30 seconds'` ordered by `created_at` ascending.
  - Per row, inside its own `step.run(`row-${row.id}`, ...)`: call `await inngest.send({ id: row.id, name: row.event_name, data: row.payload })`. On success, `UPDATE outbox SET processed_at = now() WHERE id = ? AND processed_at IS NULL`. On thrown error, `UPDATE outbox SET attempts = attempts + 1, last_error = ? WHERE id = ?` and swallow — the next sweep tick will retry. The `processed_at IS NULL` guard makes a re-fire from a competing post-commit publisher a guaranteed no-op at the DB level; Inngest's 24h dedup on the row id covers the worker side.
- `src/server/outbox/prune.ts` — exports `outboxPrune`, daily cron `0 3 * * *` (03:00 UTC, lowest-traffic). One `step.run` deleting `WHERE processed_at IS NOT NULL AND processed_at < now() - interval '7 days'`.
- `src/inngest/functions/index.ts` — replace the placeholder export with `export const functions = [outboxSweep, outboxPrune]`, importing from `~/server/outbox/sweep` and `~/server/outbox/prune`.
- `src/server/outbox/__tests__/sweep.test.ts` — rstest unit, mocked db + mocked `inngest.send`.
  - Asserts the SELECT WHERE shape (matches "≥30 s old, unprocessed").
  - Success branch: `processed_at` UPDATE issued with the row id.
  - Failure branch: `attempts` incremented and `last_error` set; no `processed_at` UPDATE.
  - Idempotent re-fire: a row with `processed_at` already set is filtered out by the SELECT, so a second sweep tick over the same data emits zero `inngest.send` calls.
- `src/server/outbox/__tests__/prune.test.ts` — rstest unit, mocked db. Asserts the DELETE WHERE clause includes the `processed_at IS NOT NULL AND processed_at < now() - interval '7 days'` shape.
- `src/server/outbox/__tests__/sweep.integration.test.ts` — rstest integration, runs only when `process.env.INTEGRATION_DB` is set (`describe.skipIf(!process.env.INTEGRATION_DB)` at the top). Connects to the Neon branch via the live `db` import, inserts 3 rows aged > 30 s (insert with `created_at: new Date(Date.now() - 60_000)`), stubs `inngest.send` to resolve, runs the sweep handler against the real db, asserts all three rows have `processed_at` set, then runs the sweep again and asserts zero further `inngest.send` calls. Cleans up its own rows in `afterAll` keyed by the unique `eventName` it inserted (CLAUDE.md isolation rule). The default unit test run skips it; developers run it locally with `INTEGRATION_DB=1 make test`.

## Phase 5: README + dev-workflow docs

### Changes

- `src/server/outbox/README.md` — new file. Sections:
  - **What this is** — one-paragraph pointer to ADR-014.
  - **Post-commit publisher pattern** — code example showing the canonical call-site shape:

    ```ts
    const id = await db.transaction(async (tx) => {
      // ... canonical state writes ...
      return outbox.publish(tx, "lead.captured", { leadId: lead.id });
    });
    try {
      await inngest.send({ id, name: "lead.captured", data: { leadId } });
    } catch (err) {
      console.error("[outbox] post-commit send failed; sweep will retry:", err);
    }
    ```

    Plus the rule: a thrown `inngest.send` must never propagate to the user. Quote ADR-014's "code-review-blocking pattern".
  - **neon-http transaction caveat** — note that `drizzle-orm/neon-http` transactions are non-interactive: queries inside the callback are batched into one HTTP request, so `.returning()` mid-callback does not surface values. Generate any id you need before the insert (which is exactly what `outbox.publish` does internally). Future call-site PRs that need a transactional `.returning()` are the trigger for evaluating a `neon-serverless` swap.
  - **Local development** — run `make start` and `make inngest_dev` in two shells; the Inngest dev UI is at `http://localhost:8288` and auto-discovers `https://www.localhost/api/inngest`.
